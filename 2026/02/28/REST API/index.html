REST API

发表于2026-02-28|更新于2026-02-28|查询示例
|浏览量:

定义

REST API (representational state transfer)，表现层状态转移。通过REST API可以对数据的状态进行增删改查。在http协议中是一种常用的接口协议。

与普通接口协议的区别

先看一种「不规范」的写法

以前很多接口是这样设计的：一个动作一个地址，动作写在 URL 里。

这样会有一大堆地址（getXxx、postXxx、deleteXxx…），难记、难维护。

用 REST 怎么改？把「动作」提前到 HTTP 方法上

REST 的做法是：地址只表示「资源」（比如文档），动作用 HTTP 方法表示。

还是「文档」这个资源，可以只用一个资源地址，用不同请求方法区分增删改查：

请求方法 地址示例 含义
GET /adm/docs 查文档列表
GET /adm/docs/1 查 id=1 的文档
POST /adm/docs 新增一篇文档
PUT /adm/docs/1 修改 id=1 的文档
DELETE /adm/docs/1 删除 id=1 的文档

所以：不是「只用一个地址实现所有接口」，而是**「一种资源一个地址（如 /adm/docs），通过 GET/POST/PUT/DELETE 区分要做的动作」**。这样地址数量少很多，语义也统一。

REST 和 SOAP、RPC、GraphQL 等一样都是接口协议，但设计思路和适用场景不同，对比如下：

协议 特点 典型场景
REST 基于 HTTP 方法（GET/POST/PUT/DELETE），资源用 URL 表示，无状态 Web API、移动端、开放平台
SOAP 基于 XML，协议重，支持 WS-* 安全/事务等 企业级、银行/政务等异构系统
RPC 远程过程调用，像调本地函数，多基于二进制 微服务内部、gRPC、Dubbo
GraphQL 单端点，客户端按需声明要的字段，减少过取/欠取 复杂前端、多端数据聚合

简单说：REST 用「URL + HTTP 方法」操作资源，SOAP 用 XML 信封发请求，RPC 直接调远程函数，GraphQL 用查询语句精确要数据。

举个例：查「id=1 的用户」

同一种需求，四种协议长这样：

REST：资源是 URL，动作是 HTTP 方法。

返回 JSON：{"id":1,"name":"张三","age":25}。
查列表用 GET /users，新建用 POST /users，改/删用 PUT /users/1、DELETE /users/1。

SOAP：所有请求都发到同一个地址，用 XML 包一层，里面写「要调什么、参数是什么」。

RPC：像在调本地函数，名字一般是「服务名.方法名」。

客户端/服务端会按约定（如 gRPC、JSON-RPC）把这句话序列化成网络包发出去。

GraphQL：只有一个端点（如 POST /graphql），请求体里写「要什么字段」。

只写 id, name 就只返回这两个字段，不会多返回别的，适合前端按需要数据。

对比一下：REST 是「一个 URL 代表一种资源，用 GET/POST 等表示动作」；SOAP/RPC 是「发请求时指名要执行哪个操作/方法」；GraphQL 是「一个入口，用查询语句描述要拿哪些数据」。

REST API 常见的方法

方法 含义 对应 SQL 说明
GET 客户端向服务器请求获取资源 SELECT 只读，不改数据，可缓存
POST 客户端向服务器提交，通常用于新增 INSERT 创建新资源、提交表单等
PUT 客户端向服务器提交，用请求体整体替换/更新资源 UPDATE 全量更新（部分更新常用 PATCH）
DELETE 客户端向服务器请求删除资源 DELETE 删除指定资源

消息体（Body）

消息体就是 HTTP 请求或响应里「真正要传的那块数据」，在协议里叫 Body。通常指的是请求信息体和接口返回的结果。

  • 请求体（Request Body）：客户端发给服务器时带上的内容。比如用 POST 新建用户时，在 body 里放 {"name":"张三","age":25}，服务器根据这段内容去插库。

  • 响应体（Response Body）：服务器返回给客户端的内容。比如 GET 某用户时，服务器把查到的数据放在响应 body 里，如 {"id":1,"name":"张三","age":25}。

  • GET：是「客户端发请求，服务器返回数据」，不是服务器主动推；且 GET 不应改数据，只做查询。所以 GET 一般没有请求体，参数用 URL 查询串传（如 GET /users?page=1&size=10）。

为啥 GET 没有请求体？客户端不是也在向服务器「发消息」吗？

客户端确实在发消息，但发的内容在「URL」里，不在 body 里。可以把一次请求想成一张单子：

  • 请求行：写的是「什么方法 + 哪个地址」，例如 GET /users/1 或 GET /users?page=2&size=10。这里已经说明了「我要查谁、查第几页」。
  • 请求体（Body）：单子下面留的一块空白，用来写「一大段数据」（比如一段 JSON）。GET 时通常不填这块——要传的条件已经写在 URL 里了，所以没有请求体，不等于没发消息，只是消息在 URL 里。
    POST/PUT 时除了 URL，还会在 body 里再带一截数据（如新建用户的姓名、年龄），服务器从 body 里读这段内容。
